# CBOS Workflow Engine Architecture

## Version 1.0

## Purpose

The Workflow Engine is the automation and orchestration layer of CBOS.

It enables organizations to define, version, execute and monitor business workflows without hardcoding industry-specific logic.

The Workflow Engine must allow CBOS to remain:

```text
Multi-Tenant
Organization Based
Project Centric
Metadata Driven
Event Driven
Industry Agnostic
```

Its purpose is to:

* react to business events
* evaluate configurable conditions
* execute controlled actions
* track every execution with auditability
* recover safely from failure
* provide a foundation for future AI-assisted automation

---

# Principles

## Principle 1

Workflows are configuration, not code.

Business automation must be expressed through metadata and versioned definitions.

---

## Principle 2

Every workflow belongs to an organization.

No workflow definition, execution or log may cross organization boundaries.

---

## Principle 3

Projects remain the operational center.

Workflows may operate on many domain entities, but the primary orchestration anchor remains the Project aggregate whenever business execution is involved.

---

## Principle 4

Everything is event driven.

Every meaningful state change may emit events, and workflows subscribe to those events through explicit triggers.

---

## Principle 5

Workflow execution must be deterministic and auditable.

Given the same workflow version, inputs and permissions, execution should be explainable step by step.

---

## Principle 6

Failure is a first-class concern.

Retries, dead letter handling and manual recovery must be built into the engine.

---

## Principle 7

The engine must be industry agnostic.

The platform may ship templates for maternity studios, marketing agencies, podcast businesses and future industries, but the engine itself cannot assume domain-specific steps.

# Workflow Scope

A workflow may be scoped at one of the following levels:

* Organization scope
* Project scope
* Entity scope
* Template scope

Examples:

* an onboarding workflow for all new clients in an organization
* a project delivery workflow for a specific project type
* an invoice reminder workflow for finance events
* a content approval workflow for a campaign or episode lifecycle

Scope is declarative and configurable. No workflow definition may embed assumptions tied to a single industry.

```text
WorkflowExecution -> WorkflowExecutionEvents (1:N)
WorkflowExecution -> WorkflowRetries (1:N)
WorkflowExecution -> DeadLetterEntries (0:N)
WorkflowExecution -> WorkflowApprovals (0:N)
Project -> WorkflowExecutions (1:N when project scoped)
```

Key relationship rules:

* every workflow definition belongs to one organization
* every execution references the exact version it ran
* published versions are immutable
* executions may reference a project, client, invoice, campaign, deliverable or other entity context
* project_id is required whenever the workflow affects operational delivery work

# Workflow Lifecycle

## Definition Lifecycle

Workflow versions move through:

```text
DRAFT -> PUBLISHED -> ARCHIVED
```

### Draft

* editable
* not executable for production events
* can be tested in isolated mode

### Published

* immutable
* active for new executions
* referenced by future triggers

### Archived

* no longer receives new executions
* retained for history and rollback

## Rollback Support

Rollback is implemented by promoting a previously published version as the new published version.

Rules:

* active executions continue on their original version
* new executions use the newly published rollback target
* rollback must create an audit event
* rollback cannot mutate historical execution records

# Execution Engine

## Core Responsibilities

The Execution Engine must:

* listen for trigger events
* resolve eligible workflow definitions by organization and scope
* load the correct published version
* evaluate conditions
* execute actions in order or branch path
* persist status, duration and retries
* handle waits, approvals and external callbacks
* emit workflow execution events

## Execution Flow

```text
Trigger Received
↓
Resolve Organization Scope
↓
Resolve Matching Workflow Definitions
↓
Load Published Version
↓
Build Execution Context
↓
Evaluate Conditions
↓
Execute Actions / Wait States / Branches
↓
Record Audit Events
↓
Complete, Retry, Pause or Dead-Letter
```

## Execution Context

Every execution context should include:

```text
organization_id
project_id
context_entity_type
context_entity_id
trigger_payload
workflow_version_id
actor_id
timezone
custom_fields
```

## Tracking Requirements

Every execution must track:

* status
* duration
* retry count
* current step
* initiating trigger
* actor
* full audit trail

# Failure Handling

## Retries

The engine must support configurable retry policies:

```text
max retry count
fixed delay
exponential backoff
retry by error class
retry by action type
```

## Dead Letter Queue

Executions that exceed retry policy or fail irrecoverably move to the dead letter queue.

Dead letter queue requirements:

* retain workflow payload
* retain error history
* retain organization and entity context
* support filtered review by admins
* support replay after correction

## Manual Intervention

Authorized users must be able to:

* retry a failed step
* skip a step where policy permits
* cancel execution
* correct input data
* approve or reject a pending action
* replay from dead letter queue

All manual actions must create audit events.

# Multi-Tenant Rules

* every workflow entity must contain organization_id where persistent tenancy applies
* workflow resolution is always organization scoped
* scheduled triggers execute in the organization timezone
* no workflow definition may reference another organization's users, projects, clients or finance records
* dead letter and audit data are tenant isolated
* workflow templates may be shared by the platform only as copied metadata, never as cross-tenant live references

---

# Security Rules

## Access Control

* only authorized users may create, publish, archive or rollback workflow versions
* execution visibility must follow organization, role, team and entity access rules
* manual triggers and intervention actions require explicit permissions

## Sensitive Actions

The following categories require additional protection:

* finance mutations
* approvals
* external communications
* API callbacks that alter business state

Controls:

* role-based permissions
* approval gates
* full audit logging
* least-privilege service identities

## Data Protection

* all payloads must be organization scoped
* secrets for outbound integrations must not be stored in workflow definitions in plain text
* personally identifiable data in logs must follow masking policy where required

## AI Safety

AI may suggest workflows, optimizations and bottlenecks.

AI may not autonomously publish workflows, send payments, issue refunds or bypass approval rules.

# Scaling Strategy

## Architectural Approach

The engine should scale through asynchronous and distributed execution.

## Scaling Components

### Event Ingestion Layer

Receives domain events, schedule events, manual events and API events.

### Workflow Resolver

Determines which workflow definitions apply for the incoming event.

### Execution Queue

Buffers workflow executions for asynchronous processing.

### Worker Pool

Processes workflow actions in parallel across organizations while preserving execution order within a single workflow instance where needed.

### State Store

Persists definitions, versions, executions, retries and audit events.

### Dead Letter Queue

Separates unrecoverable failures for operational review.

## Scaling Rules

* partition queues by organization_id and event class where useful
* use idempotency keys for API and event reprocessing
* allow horizontal scaling of workers
* isolate long-running wait states from active compute workers
* archive historical execution data without losing auditability
* optimize reads for operational dashboards and audits separately

# Industry-Agnostic Validation

## Can this run a maternity studio?

Yes. The engine supports session booking, reminders, shoot preparation, proofing, invoicing and delivery through configurable triggers, conditions and actions without maternity-specific hardcoding.

## Can this run a marketing agency?

Yes. The same engine supports campaign approvals, task routing, client communication, milestone tracking, billing and publishing workflows through metadata and project context.

## Can this run a podcast business?

Yes. The same model supports episode planning, guest coordination, recording, editing, approval, publishing and sponsor invoicing using configurable events and actions.

## Constitutional Test

If a new business requires only new workflow definitions, statuses, fields, templates or permissions, the engine is sufficiently industry agnostic.

If a new business requires new hardcoded workflow logic in the engine, the architecture has violated CBOS principles.

---

# Constitutional Rule

The Workflow Engine shall orchestrate business behavior through tenant-owned metadata, versioned workflow definitions, event-driven execution and auditable actions, while remaining project-centric and industry-extensible.
